Module 12: Professional Research Repositories

Build a publication-ready repository with README, LICENSE, semantic versioning, and a pre-publication checklist.

Overview

A professional research repository is one that a colleague — or a reviewer — can clone, run, and understand without asking you questions. This module covers the structure, documentation, and release process that transform a working Git repository into a citable, reproducible research artifact.

Prerequisites

  • Module 03 — Remote Repositories
  • Module 08 — Data Management (data safety practices)

Research Repository Structure

A well-structured repository signals professionalism and aids reproducibility:

tree ~/rehab-research
rehab-research/
├── README.md               ← project overview, installation, usage, citation
├── LICENSE                 ← how others may use your code and data
├── .gitignore              ← excludes data/, credentials, system files
├── renv.lock               ← (R projects) locked package versions
├── scripts/
│   ├── analysis.R          ← main analysis functions
│   ├── figures.R           ← figure generation
│   └── preprocessing.R     ← data cleaning (operates on data/ locally)
├── docs/
│   ├── protocol.md         ← study protocol and pre-registration details
│   └── codebook.md         ← variable definitions and measurement descriptions
├── figures/                ← gitignored — generated by scripts/figures.R
│   └── .gitkeep            ← empty file to preserve directory in repository
└── data/                   ← gitignored — never committed
    └── .gitkeep            ← preserves directory structure for collaborators

Why .gitkeep? An empty directory cannot be committed (Git tracks files, not directories). An empty .gitkeep file preserves the directory structure so collaborators know where to put their local data files after cloning.

touch figures/.gitkeep data/.gitkeep
git add figures/.gitkeep data/.gitkeep
git commit -m "Structure: add gitkeep files to preserve empty directories"

Writing README.md from the Terminal

A README answers five questions any reader will have:

nano README.md

Template (fill in for your project):

# Grip Strength and Balance Outcomes in Stroke Rehabilitation

Analysis code for the study: "Recovery trajectories of grip strength and postural
balance in adults following first-time stroke." [Journal, Year, DOI when available]

## Installation

```bash
git clone git@github.com:yourusername/rehab-research.git
cd rehab-research
Rscript -e "renv::restore()"   # installs all R packages from renv.lock

Usage

Data files are not included in this repository (participant privacy). Place your data files in data/ following the codebook in docs/codebook.md.

Run the analysis:

Rscript scripts/preprocessing.R   # cleans raw data → data/processed/
Rscript scripts/analysis.R        # runs main analysis → output/results.csv
Rscript scripts/figures.R         # generates figures → figures/

Data

Raw data are archived at [Repository Name] under accession number [XXXXX]. Contact: j.bayer@rehab-institute.edu

Citation

Bayer, J. et al. (2026). Recovery trajectories of grip strength and postural balance in adults following first-time stroke. Journal Name, XX(X), XXX–XXX. https://doi.org/10.XXXX/XXXXX

License

Code: MIT License (see LICENSE file). Data: CC BY 4.0 (see data archive).


Save and commit:

```bash
git add README.md
git commit -m "Docs: add README with installation, usage, and citation"

Choosing a License

Research code and content have different licensing norms:

License Best for Key permission
MIT Analysis scripts, R functions, Python code Use, modify, distribute freely; attribution required
Apache 2.0 Software tools with patent considerations Same as MIT + explicit patent grant
GPL-3.0 Code built on GPL-licensed tools Derivatives must also be open source
CC BY 4.0 Documentation, data, figures Use freely with attribution
CC BY-NC 4.0 Content you want to restrict commercially Non-commercial use only

For most analysis code, MIT is the appropriate choice. Add it:

cat > LICENSE << 'EOF'
MIT License

Copyright (c) 2026 Dr. J. Bayer

Permission is hereby granted, free of charge, to any person obtaining a copy
of this software and associated documentation files (the "Software"), to deal
in the Software without restriction, including without limitation the rights
to use, copy, modify, merge, publish, distribute, sublicense, and/or sell
copies of the Software, and to permit persons to whom the Software is
furnished to do so, subject to the following conditions:

The above copyright notice and this permission notice shall be included in all
copies or substantial portions of the Software.

THE SOFTWARE IS PROVIDED "AS IS", WITHOUT WARRANTY OF ANY KIND, EXPRESS OR
IMPLIED, INCLUDING BUT NOT LIMITED TO THE WARRANTIES OF MERCHANTABILITY,
FITNESS FOR A PARTICULAR PURPOSE AND NONINFRINGEMENT. IN NO EVENT SHALL THE
AUTHORS OR COPYRIGHT HOLDERS BE LIABLE FOR ANY CLAIM, DAMAGES OR OTHER
LIABILITY, WHETHER IN AN ACTION OF CONTRACT, TORT OR OTHERWISE, ARISING FROM,
OUT OF OR IN CONNECTION WITH THE SOFTWARE OR THE USE OR OTHER DEALINGS IN THE
SOFTWARE.
EOF

git add LICENSE
git commit -m "Legal: add MIT license for analysis code"

Semantic Versioning via git tag

Version your analysis code using MAJOR.MINOR.PATCH conventions:

Version bump When Example change
MAJOR Breaking change to analysis methodology Switch from min-max to z-score normalization
MINOR New measure or function added Add grip strength asymmetry index
PATCH Bug fix, clarification, typo Fix variable name in codebook

Tag the first public release:

# Final cleanup before tagging
git log --oneline
git status
On branch main
nothing to commit, working tree clean
git tag -a v1.0.0 -m "Analysis for Wave 1 manuscript submission (Smith et al., 2026)"

Verify:

git show v1.0.0
tag v1.0.0
Tagger: Dr. J. Bayer <j.bayer@rehab-institute.edu>
Date:   Fri Feb 28 16:00:00 2026 +0100

Analysis for Wave 1 manuscript submission (Smith et al., 2026)

commit 3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c2 (HEAD -> main, tag: v1.0.0)
...

Push the tag:

git push origin v1.0.0
Total 0 (delta 0), reused 0 (delta 0), pack-reused 0
To git@github.com:yourusername/rehab-research.git
 * [new tag]         v1.0.0 -> v1.0.0

Archiving for Publication: git archive

Create a clean snapshot of the repository at a specific tag — no .git/ history, no ignored files — as a zip archive for manuscript submission or data repository deposit:

git archive --format=zip v1.0.0 > rehab-research-v1.0.0.zip

Verify the contents:

unzip -l rehab-research-v1.0.0.zip | head -20
Archive:  rehab-research-v1.0.0.zip
  Length      Date    Time    Name
---------  ---------- -----   ----
      523  02-28-2026 16:00   README.md
     1084  02-28-2026 16:00   LICENSE
       89  02-28-2026 16:00   .gitignore
     2341  02-28-2026 16:00   scripts/analysis.R
     1102  02-28-2026 16:00   scripts/figures.R
      467  02-28-2026 15:30   docs/protocol.md
...

The archive contains only committed, tracked files at the v1.0.0 state. Data files in data/ are not included (they are gitignored).

Creating a GitHub Release

Make the tag visible as a formal release on GitHub:

gh release create v1.0.0 \
  --title "Wave 1 Analysis — Manuscript Submission" \
  --notes "Analysis code accompanying: Bayer et al. (2026) *Journal Name*.

**What's included**: All analysis scripts, codebook, and protocol documentation.

**Data**: Raw participant data are not included per IRB data management plan.
Request access through the institutional data archive (DOI: XXXXX)." \
  rehab-research-v1.0.0.zip
https://github.com/yourusername/rehab-research/releases/tag/v1.0.0

The release page on GitHub now shows the release notes and the zip archive as a downloadable asset. This URL is citable and persistent.

Publication Readiness Terminal Checklist

Run all eight checks before pushing to a public remote or submitting to a journal:

# 1. Check repository size
echo "=== Repository size ===" && git count-objects -vH

# 2. Confirm .gitignore excludes data directory
echo "=== .gitignore data check ===" && grep "^data" .gitignore && echo "PASS: data/ excluded" || echo "FAIL: no data/ pattern found"

# 3. List all currently tracked files (check for unexpected data files)
echo "=== All tracked files ===" && git ls-files

# 4. Search history for any CSV files ever committed
echo "=== CSV history ===" && git log --all --full-history -- "*.csv" | head -5

# 5. Check for any .env files ever committed
echo "=== .env history ===" && git log --all --full-history -- ".env" | head -5

# 6. Verify README exists and is committed
echo "=== README check ===" && git ls-files README.md && echo "PASS" || echo "FAIL: no README.md"

# 7. Verify LICENSE exists and is committed
echo "=== LICENSE check ===" && git ls-files LICENSE && echo "PASS" || echo "FAIL: no LICENSE"

# 8. Verify version tag exists
echo "=== Tags ===" && git tag

All eight commands should return “PASS” or expected content. Any failures indicate a gap to address before publication.
