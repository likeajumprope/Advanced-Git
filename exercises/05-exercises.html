Exercises: Module 05 — Merging and Conflicts

Before You Begin

Complete Module 05 and Exercises 04. You should have at least two branches in ~/rehab-research/.

Exercise 5.1 — Create and Resolve a Merge Conflict

Goal: Deliberately create a conflict between two branches, then resolve it using nano.

Setup:

  1. Make sure you are on main:

    cd ~/rehab-research
git switch main

  2. Modify line 3 of analysis.R on main:

    # View current content
cat -n scripts/analysis.R

    Edit with nano and change the normalize_balance function body:

    nano scripts/analysis.R

    Change the normalize function to:

    normalize_balance <- function(x) (x - mean(x)) / sd(x)

    Save: Ctrl+O, Return, Ctrl+X

    git add scripts/analysis.R
git commit -m "Fix: use z-score normalization for balance index"

  3. Switch to feature/add-grip-strength-measure and make a DIFFERENT change to the same line:

    git switch feature/add-grip-strength-measure
nano scripts/analysis.R

    Change the normalize function to:

    normalize_balance <- function(x) (x - min(x)) / (max(x) - min(x)) * 100

    Save and commit:

    git add scripts/analysis.R
git commit -m "Fix: use min-max normalization (0-100 range) for balance"

Now create the conflict:

  1. Switch to main and attempt to merge:

    git switch main
git merge feature/add-grip-strength-measure
Expected Output (conflict) 
Auto-merging scripts/analysis.R
CONFLICT (content): Merge conflict in scripts/analysis.R
Automatic merge failed; fix conflicts and then commit the result.

  1. Check the conflict status:

    git status
Expected Output 
On branch main
You have unmerged paths.
  (fix conflicts and run "git merge --continue")
  (use "git merge --abort" to abort the merge)

Unmerged paths:
  (use "git add <file>..." to mark resolution)
        both modified:   scripts/analysis.R

  1. View the conflict markers:

    cat scripts/analysis.R
Expected Output 
library(dplyr)
library(tidyr)

<<<<<<< HEAD
normalize_balance <- function(x) (x - mean(x)) / sd(x)
=======
normalize_balance <- function(x) (x - min(x)) / (max(x) - min(x)) * 100
>>>>>>> feature/add-grip-strength-measure

filter_complete_cases <- function(df) dplyr::filter(df, !is.na(grip_strength))
measure_grip_strength <- function(hand, trials) mean(trials[hand])

HEAD version = z-score (your main branch). Feature version = min-max (incoming branch).

  1. Resolve the conflict using nano:

    nano scripts/analysis.R

    Delete the three marker lines and the z-score version, keeping the min-max version:

    normalize_balance <- function(x) (x - min(x)) / (max(x) - min(x)) * 100

    Save and exit.

  2. Verify no conflict markers remain:

    cat scripts/analysis.R

    Confirm: no <<<<<<<, =======, or >>>>>>> in the file.

  3. Complete the merge:

    git add scripts/analysis.R
git status
Expected Output 
On branch main
All conflicts fixed but you are still merging.
  (use "git merge --continue" to continue)
  (use "git merge --abort" to abort the merge)

Changes to be committed:
        modified:   scripts/analysis.R
git merge --continue

Nano opens for the merge commit message — save the default and exit.

  1. Verify the merge:

    git log --oneline --graph --all
Expected Output 
*   a7b8c9d (HEAD -> main) Merge branch 'feature/add-grip-strength-measure'
|\
| * 3c4d5e6 Fix: use min-max normalization for balance
* | 9f0a1b2 Fix: use z-score normalization for balance index
|/
* 1a5f3b2 Add: filter_complete_cases to exclude missing values

The diamond shape shows the divergence and merge. Both lines of work combined.

What you learned:

  • Conflict markers (<<<<<<<, =======, >>>>>>>) show both versions
  • You must delete ALL three markers and choose the content you want
  • git add after editing marks the conflict as resolved
  • git merge --continue completes the merge
Common Mistakes
  • Left conflict markers in the file: git add will succeed, but the file will be broken. Always cat the file to verify before staging.
  • Used git commit instead of git merge --continue: Both work when in a merge state, but git merge --continue is more explicit about your intent.

Exercise 5.2 — Abort a Merge

Goal: Practice aborting a merge and returning to a clean state.

Setup: Create another conflict situation.

Steps:

  1. Create a new branch and modify analysis.R:

    git switch -c fix/alternative-filter
echo "" >> scripts/analysis.R
echo "filter_by_age <- function(df, min_age) dplyr::filter(df, age >= min_age)" >> scripts/analysis.R
git add scripts/analysis.R
git commit -m "Add: filter_by_age function for age-based inclusion criteria"

  2. Modify the same lines on main:

    git switch main
echo "" >> scripts/analysis.R
echo "filter_by_age <- function(df, max_age) dplyr::filter(df, age <= max_age)" >> scripts/analysis.R
git add scripts/analysis.R
git commit -m "Add: filter_by_age with upper age limit for exclusion"

  3. Attempt the merge:

    git merge fix/alternative-filter
Expected Output 
Auto-merging scripts/analysis.R
CONFLICT (content): Merge conflict in scripts/analysis.R
Automatic merge failed; fix conflicts and then commit the result.

  1. Decide you are not ready — abort:

    git merge --abort

  2. Verify the working directory is clean:

    git status
cat scripts/analysis.R
Expected Output of git status 
On branch main
nothing to commit, working tree clean

The file shows your main version with no conflict markers. The merge never happened — you are back where you started.

What you learned:

  • git merge --abort completely reverses the merge attempt
  • You can safely abort at any point before git merge --continue
  • Aborting does not affect any commits — only the in-progress merge

Summary

You can now:

  • Merge branches with git merge
  • Identify conflict markers in a conflicted file
  • Resolve conflicts using nano and git add
  • Complete a merge with git merge --continue
  • Abort a merge with git merge --abort

You are ready for Module 06 — Rebase.
