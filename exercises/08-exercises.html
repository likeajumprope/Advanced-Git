Exercises: Module 08 — Data Management

Before You Begin

Complete Module 08 and Exercises 02. You should have a ~/rehab-research/ repository with a clean working tree.

Data Safety Checkpoint

Before proceeding: run git status and confirm you have no uncommitted data files. These exercises create files that simulate participant data — they will be cleaned up by .gitignore configurations you write.

Exercise 8.1 — Configure .gitignore for a Research Project

Goal: Create realistic research data files and configure .gitignore to exclude them.

Steps:

  1. Navigate to your repository and create simulated data files:

    cd ~/rehab-research
mkdir -p data
echo "id,grip_kg,balance_score" > data/results.csv
echo "1001,28.4,72.3" >> data/results.csv
echo "SECRET_KEY=abc123" > .env
echo ".RData" > .RData

  2. Check the current status — all three files appear as untracked:

    git status
Expected Output 
On branch main
Untracked files:
  (use "git add <file>..." to include in what will be committed)
        .RData
        .env
        data/

nothing added to commit but untracked files present (use "git add" to track)

All three are visible — Git does not know to exclude them yet.

  1. Create .gitignore with research-appropriate patterns:

    nano .gitignore

    Add the following lines:

    data/
.env
*.RData

    Save: Ctrl+O, Return, Ctrl+X

  2. Check status again:

    git status
Expected Output 
On branch main
Untracked files:
  (use "git add <file>..." to include in what will be committed)
        .gitignore

nothing added to commit but untracked files present (use "git add" to track)

data/, .env, and .RData no longer appear. Only .gitignore itself is untracked.

  1. Commit the .gitignore:

    git add .gitignore
git commit -m "Data: add .gitignore to protect participant data and credentials"
Common Mistakes
  • Forgetting the trailing / on directory names: data (no slash) matches a file named data OR a directory. data/ explicitly targets the directory. Both work, but data/ is clearer.
  • Path separator on Windows: .gitignore uses forward slashes (data/) even on Windows with Git Bash. Backslashes (data\) do not work in .gitignore.
  • Quoting the pattern: echo "data/" >> .gitignore is correct. Do not add quotes inside the file itself — "data/" (with quotes) would match a directory literally named "data/" (including quote characters).

Exercise 8.2 — Remove an Accidentally Committed File

Goal: Simulate the common mistake of committing data before .gitignore is configured, then correctly remove it from version control while keeping it on disk.

Setup: First, create a situation where data was committed accidentally.

  1. Temporarily remove .gitignore for this simulation:

    git rm .gitignore
git commit -m "Temp: remove gitignore (simulation only)"

  2. Now commit a data file (simulating the mistake):

    echo "id,name,dob" > data/participants.csv
echo "1001,Smith,1985-03-14" >> data/participants.csv
git add data/participants.csv
git commit -m "Add: participant data file"

  3. Verify the file is tracked:

    git log --oneline -- data/participants.csv
Expected Output 
a1b2c3d Add: participant data file

The file appears in history — this is the problem we are solving.

  1. Remove from version control (but keep on disk):

    git rm --cached data/participants.csv
Expected Output 
rm 'data/participants.csv'

  1. Verify the file still exists on disk:

    ls data/
Expected Output 
participants.csv

The file is still there — --cached only removed it from Git’s index.

  1. Re-add .gitignore with data/ pattern:

    nano .gitignore

    Add:

    data/
.env
*.RData

    Save and exit.

  2. Check status and commit:

    git status
Expected Output 
On branch main
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
        deleted:    data/participants.csv

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        .gitignore

data/participants.csv is staged for deletion from version control. .gitignore is new and untracked.

git add .gitignore
git commit -m "Data: remove participant data from version control tracking"

  1. Confirm the file is gone from current tracking but still on disk:

    git ls-files data/
ls data/
Expected Output 
(empty — git ls-files shows nothing in data/)

participants.csv

git ls-files lists only files Git is tracking — data/participants.csv is no longer tracked. The ls data/ confirms it is still physically present on disk.

The Commit Is Still in History

Even after this exercise, data/participants.csv exists in the commit "Add: participant data file". Anyone with access to this repository can run git show on that hash and read the file content.

In a real scenario with actual PHI: immediately contact your institution’s data security office. Removing from history requires git filter-repo and force-pushing — actions that affect every collaborator and require coordination.

Exercise 8.3 — Audit Repository Size and Content

Goal: Use Git inspection commands to understand what is stored in your repository and identify potential problems before publication.

Steps:

  1. Check the repository size:

    cd ~/rehab-research
git count-objects -vH
Expected Output 
count: 28
size: 89.50 KiB
in-pack: 0
packs: 0
size-pack: 0 bytes
prune-packable: 0
garbage: 0
size-garbage: 0 bytes

count is the number of loose objects. size is their total disk usage. Your numbers will differ depending on how many commits you have made. For a code-only research repository, this should stay under a few hundred KiB.

  1. Search history for any CSV files that were ever committed:

    git log --all --full-history -- "*.csv"
Expected Output (if you followed Exercise 8.2) 
a1b2c3d Add: participant data file

This shows the commit where data/participants.csv was accidentally committed. Even though it is no longer tracked, it remains in history.

If this output is empty, no CSV was ever committed — your repository is clean.

  1. List all currently tracked files:

    git ls-files
Expected Output 
.gitignore
scripts/analysis.R

Only files that Git is actively tracking appear. If you see anything in data/, that file is still tracked and needs git rm --cached treatment.

  1. Check for common PHI-related filename patterns in tracked files:

    git ls-files | grep -iE "participant|subject|patient|dob|phi" && echo "FOUND potential PHI filenames" || echo "No PHI-related filenames found"
Expected Output (after Exercise 8.2) 
No PHI-related filenames found

The grep returns nothing because participants.csv is no longer tracked. The || (OR operator) runs the echo when grep finds no matches.

  1. Verify .gitignore is correctly excluding the data directory:

    git check-ignore -v data/participants.csv
Expected Output 
.gitignore:1:data/  data/participants.csv

This shows: file .gitignore, line 1, pattern data/ is ignoring data/participants.csv. The git check-ignore -v command explains exactly which .gitignore rule is responsible for ignoring a file — useful for debugging unexpected ignore behavior.

What you learned:

  • .gitignore prevents files from being staged — but only if they are not already tracked
  • git rm --cached removes a file from tracking while keeping it on disk
  • Past commits retain file content even after .gitignore is configured
  • git count-objects -vH, git ls-files, and git log --all -- pattern give you a complete picture of what is stored in your repository
Common Mistakes
  • Using git rm without --cached: This deletes the file from disk AND stops tracking it. Always use --cached when you want to keep the file.
  • Running git add . after setting up .gitignore: Generally safe once .gitignore is in place, but double-check git status output before committing to confirm no sensitive files slipped through.
  • Assuming .gitignore fixes past commits: It does not. History is permanent until you take explicit destructive action with git filter-repo.

Summary

You can now:

  • Create .gitignore files from the terminal using nano, echo, or heredoc
  • Identify and use research-specific exclusion patterns
  • Remove accidentally committed files with git rm --cached
  • Audit repository content and size with git ls-files, git log --all, and git count-objects
  • Explain why .gitignore does not retroactively protect past commits
