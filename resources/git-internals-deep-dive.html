Git Internals Deep Dive

Supplementary material exploring Git’s object model, pack files, refs, and plumbing commands.

This page expands on Module 01 — Git Concepts for learners who want to understand Git’s internal architecture. No new workflow commands are introduced — this is pure understanding of the mechanics beneath the commands you already know.

Blob Object Format

Every file stored in Git is represented as a blob object — a compressed file in .git/objects/ whose name is the SHA-1 hash of its content.

Inspect the type and content of any object:

# Find a recent commit's tree hash
git log --oneline -1
3c4d5e6 Fix: use z-score normalization for balance index
# What type of object is this commit hash?
git cat-file -t 3c4d5e6
commit
# What does the commit object contain?
git cat-file -p 3c4d5e6
tree 7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c2d3e4f5a6b
parent 9d4c7e1a8b5c6d7e8f9a0b1c2d3e4f5a6b7c8d9
author Dr. J. Bayer <j.bayer@rehab-institute.edu> 1709129391 +0100
committer Dr. J. Bayer <j.bayer@rehab-institute.edu> 1709129391 +0100

Fix: use z-score normalization for balance index

The commit points to a tree object — let’s inspect it:

git cat-file -p 7a8b9c0d
100644 blob 4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3f    .gitignore
100644 blob 1a2b3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b    README.md
040000 tree 2b3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c    scripts

The tree lists filenames, permissions, and the hash of each blob or subtree. Follow the scripts tree:

git cat-file -p 2b3c4d5e
100644 blob 9c0d1e2f3a4b5c6d7e8f9a0b1c2d3e4f5a6b7c8d    analysis.R

And inspect the blob:

git cat-file -p 9c0d1e2f
library(dplyr)
library(tidyr)

normalize_balance <- function(x) (x - mean(x)) / sd(x)
filter_complete_cases <- function(df) dplyr::filter(df, !is.na(grip_strength))

The full file content, stored compressed. This is the entirety of what Git keeps for each version of a file — no metadata, just content, addressed by hash.

Tree Objects

A tree object is a directory listing. The format of each entry is:

MODE TYPE HASH    FILENAME
Mode Meaning
100644 Regular file
100755 Executable file
120000 Symbolic link
040000 Subdirectory (tree)

List the current HEAD tree directly:

git ls-tree HEAD
100644 blob 4e5f6a7b .gitignore
100644 blob 1a2b3c4d README.md
040000 tree 2b3c4d5e scripts

Recurse into subdirectories:

git ls-tree -r HEAD
100644 blob 4e5f6a7b .gitignore
100644 blob 1a2b3c4d README.md
100644 blob 9c0d1e2f scripts/analysis.R

-r expands all subtrees to show every file with its full path.

Pack Files

Loose objects (individual files in .git/objects/XX/) are efficient for fast writes but inefficient for storage and transfer. Git periodically packs objects:

# Check current loose object count
git count-objects
count: 42
size: 84 kilobytes
# Force pack manually (normally automatic)
git gc
Enumerating objects: 36, done.
Counting objects: 100% (36/36), done.
Delta compression using up to 8 threads
Compressing objects: 100% (28/28), done.
Writing objects: 100% (36/36), done.
Total 36 (delta 8), reused 0 (delta 0), pack-reused 0
# See what was created
ls .git/objects/pack/
pack-7f3a1b2c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f.idx
pack-7f3a1b2c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f.pack

Two files: .pack (the compressed objects) and .idx (an index for fast lookup). Pack files use delta compression — similar files are stored as a base + list of differences, dramatically reducing storage for versioned text files.

# Inspect pack contents
git verify-pack -v .git/objects/pack/*.idx | head -10
9c0d1e2f blob   127 89 1234
1a2b3c4d blob   512 201 1323
...

Columns: hash, type, uncompressed size, packed size, offset in pack file.

Refs in Depth

Branches are text files in .git/refs/heads/:

cat .git/refs/heads/main
3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c2d

Tags are in .git/refs/tags/:

cat .git/refs/tags/v1.0.0
9f0a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a

For annotated tags, this hash points to a tag object (not directly to a commit):

git cat-file -t 9f0a1b2c
tag
git cat-file -p 9f0a1b2c
object 3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c2d
type commit
tag v1.0.0
tagger Dr. J. Bayer <j.bayer@rehab-institute.edu> 1709129391 +0100

Analysis for Wave 1 manuscript submission

Packed refs: When there are many refs, Git packs them into .git/packed-refs for efficiency:

cat .git/packed-refs
# pack-refs with: peeled fully-peeled sorted
3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c2d refs/heads/main
9f0a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a refs/tags/v1.0.0

The reflog file:

cat .git/logs/HEAD | head -5
0000000000000000000000000000000000000000 3f2a1b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0a Dr. J. Bayer <j.bayer@rehab-institute.edu> 1709042800 +0100  commit (initial): Add: initial analysis script
3f2a1b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0a 9d4c7e1a8b5c6d7e8f9a0b1c2d3e4f5a6b7c8d9 Dr. J. Bayer <j.bayer@rehab-institute.edu> 1709086400 +0100  commit: Add: normalize_balance function

Each line: old-hash new-hash identity timestamp message. The all-zeros hash (000...) represents “nothing” — used for the initial commit.

Plumbing vs Porcelain Commands

Git has two layers of commands:

  • Porcelain: High-level, user-friendly commands (git add, git commit, git log)
  • Plumbing: Low-level, scriptable commands that operate on Git internals
Porcelain Plumbing equivalent
git add FILE git hash-object -w FILE + git update-index --add --cacheinfo ...
git commit -m "msg" git write-tree + git commit-tree + git update-ref
git log git rev-list HEAD + git cat-file -p HASH per commit
git diff git diff-tree or git diff-index
git branch git for-each-ref refs/heads/
git merge git merge-base + git read-tree + git merge-index
git push git pack-objects + git receive-pack (remote)
git show HASH git cat-file -p HASH

Plumbing commands are stable across Git versions (their output format is guaranteed not to change). Scripts that parse Git output should use plumbing commands. Porcelain output can change in formatting between Git versions.

Example: List all commit hashes reachable from HEAD:

git rev-list HEAD
3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c2d
9d4c7e1a8b5c6d7e8f9a0b1c2d3e4f5a6b7c8d9
3f2a1b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0a

Example: Find all objects in the repository and their types:

git rev-list --objects --all | head -10
3c4d5e6f commit
7a8b9c0d tree
9c0d1e2f blob    scripts/analysis.R
4e5f6a7b blob    .gitignore

How a Commit Is Created Internally

When you run git commit -m "message", Git performs these plumbing operations:

  1. Write blobs: For each staged file, write its content as a blob object (git hash-object -w FILE)

  2. Write tree: Create a tree object from the staging area’s current state (git write-tree) — returns the tree hash

  3. Write commit: Create a commit object pointing to the tree, the parent commit, the author/committer, and the message (git commit-tree TREE -p PARENT -m "message")

  4. Update branch ref: Write the new commit hash to the current branch file (git update-ref refs/heads/main NEW_HASH)

  5. Update HEAD: HEAD already points to refs/heads/main — no change needed

The entire sequence adds a handful of compressed files to .git/objects/ and updates one text file in .git/refs/heads/.
